## 2. 背景与痛点 (Background & Pain Points)
- **所属科研领域**：跨学科通用 AI 辅助（覆盖域自适应、太赫兹、医学图像处理）
- **现有流程描述**：在当前人工智能与交叉学科研究爆发式增长的背景下，科研人员面临着从“信息匮乏”到“信息过载”的剧烈转变。团队成员虽在不同领域（域自适应、太赫兹、医学图像），但均深陷于每日繁琐的文献检索工作中。
- **存在的痛点**：
  - **高认知负荷与低转化率**：每日需手动筛选 ArXiv 涌现的数百篇论文。仅通过 Title 和 Abstract 难以快速穿透其核心创新点，导致大量时间浪费在低相关性文献的阅读上。
  - **通用 LLM 的“幻觉”信任危机**：目前的通用大模型在推荐论文时，常会出现**虚构标题**或**错误匹配作者**的问题。缺乏可追溯的来源（Source Tracing）使得模型给出的建议难以直接支撑科研决策。
  - **跨领域协同的孤岛效应**：现有工具难以平衡“通用搜索”与“极度垂直”的矛盾。团队急需一个既能并行处理多领域需求，又能根据个人研究品味进行“加权排序”的智能化工具。


## 3. 大模型解决方案 (LLM-based Solution)
- **应用路径**：路径 B: 辅助科研工具
- **核心技术选型**：
  - **模型基座：**Qwen-Max
  - **技术手段：**RAG(检索增强) 、Agent(智能体)
  - **向量数据库：**引入 **Milvus 或 ChromaDB**，用于持久化存储用户的“科研画像”（研究方向、关注偏好）以及已处理文献的摘要向量 
- **跨领域文献雷达 Agent逻辑架构图**： 
    - **1. 数据采集与预处理层：**
        - 多源采集：通过 API 获取 `ArXiv`、`PubMed` (医学) 及` IEEE `(太赫兹) 的文献数据 。
        - 特征提取：提取标题、摘要及 DOI 等元数据 。
    - **2. RAG 检索增强层 ：**
        - **Embedding 转化**：使用嵌入模型将新论文转化为向量。
        - **相似度加权**：系统从向量数据库中检索用户的历史偏好（如“域自适应中的 GMM 像素级风格迁移”相关背景），计算新论文与科研画像的关联度。
    - **3. 文献雷达智能体核心 (Agent Core) :**
        - **相关性过滤**：基于 RAG 检索结果进行 Top 10 粗筛 。
        - **深度精读总结**：调用 **Qwen-Max**结合系统提示词（三维评分体系）生成摘要 。
        - **可信度校验**：启动 **DOI 溯源插件**，通过直接请求学术数据库接口检查文献真实性，消除 LLM 幻觉 
    - **4. 个性化输出层**：
        - 按用户维度（如域自适应方向)生成专属 Markdown 简报，并支持邮件或可视化门户推送 。

## 4. 数据与资源 (Data & Resources)
- **数据集/文献来源**： 
  - **计算机视觉/通用 AI**：利用 **ArXiv API** 获取每日首发论文 。
  - **医学图像处理**：接入 **PubMed API**，确保覆盖医学影像相关的核心临床与技术文献 。
  - **太赫兹/电子硬件**：通过 **IEEE Xplore API**（或 Semantic Scholar 开放接口）补充电子工程与物理层面的高质量期刊文献 。

- **算力需求**：
  - **云端算力**：核心逻辑通过调用 **Qwen-2.5-Plus API** 实现，无需本地 GPU 资源 。
  - **本地环境**：仅需轻量级服务器（或个人电脑）运行 Python 调度脚本及向量数据库（如 ChromaDB） 。

- **存储资源**：
  - **向量知识库**：本地存储经过 Embedding 处理的文献摘要与用户科研画像，预计初始占用空间小于 10GB。

## 5. 预期成果与评测 (Expected Outcomes & Evaluation)
- **预期成果**：

  - **文献雷达可视化门户**：基于 **Streamlit** 构建的 Web 界面，支持用户在线编辑科研画像（关键词、关注学者）并查看历史推送记录 。
  - **多模态个性化简报**：每日 8:00 定时推送 。简报除基础摘要外，新增由 Agent 评估的“代码复现可行性评分”与“创新点冲突检测” 。
  - **开源自动化流水线**：一套完整的 Python 工作流，包含多源 API 抓取、RAG 向量检索与多通道推送（邮件/即时通讯工具） 。

- **评测指标**：

  - **技术指标 (System Performance)**

      | 指标名称       | 定义/计算公式                                        | 目标值    | 验证方式                 |
      | :------------- | :--------------------------------------------------- | :-------- | :----------------------- |
      | **推送准时率** | 实际发送时间在8:00-8:15之间的天数 / 总天数           | **> 98%** | 连续30天运行日志自动统计 |
      | **系统可用性** | (总天数 - 爬虫/邮件服务故障天数) / 总天数            | **> 99%** | 连续30天运行日志自动统计 |
      | **数据覆盖率** | 爬虫抓取到的该领域论文数 / arXiv该领域实际新增论文数 | **> 95%** | arXiv API交叉验证        |
      | **解析成功率** | 成功提取标题、摘要、作者信息的论文数 / 抓取总数      | **> 99%** | 连续30天运行日志自动统计 |
  
  - **内容质量指标 (Content Quality)**

    | 指标名称                  | 定义/计算公式                                       | 目标值          | 验证方式                                                     |
    | :------------------------ | :-------------------------------------------------- | :-------------- | :----------------------------------------------------------- |
    | **准确率 (Precision@10)** | 用户标记为有用的论文数 / 10篇                       | **> 80%**       | 用户周度反馈（4分以上算有用）                                |
    | **召回率 (Recall@K)**     | 系统发现的重要论文数 / 该领域当天重要论文总数       | **> 70%**       | **基准集法**：以 Hugging Face Daily Papers 或各领域顶刊官网热点作为“真值表”进行交叉比对 |

    | **领域相关性**            | 主观判断属于研究领域的论文数 / 10篇                 | **> 80%**       | 用户周度自主反馈                                             |
